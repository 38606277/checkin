# "死了么"类 APP 设计文档（Technical Design Doc）

## 1. 设计目标
- 支撑高频“报平安”写入与准实时提醒。
- 在社交互动与高优先级通知之间进行链路隔离。
- 满足可观测性、可扩展性与合规可审计要求。

---

## 2. 总体架构

## 2.1 架构分层
1. **客户端层（iOS/Android）**
   - 展示首页倒计时、动态流、设置页。
   - 承担本地缓存、弱网重试、推送点击路由。
2. **网关层（API Gateway）**
   - 统一鉴权、限流、灰度路由、设备指纹注入。
3. **业务服务层（微服务）**
   - 用户服务（账号、好友、黑名单）
   - Check-in 服务（报平安核心）
   - Reminder 服务（超时计算、任务调度）
   - Feed 服务（动态流聚合）
   - Notification 服务（Push/SMS/Email）
   - Risk 服务（反骚扰、举报审核）
4. **数据与中间件层**
   - MySQL（事务数据）
   - Redis（计时器、会话、热点缓存）
   - Kafka/RabbitMQ（异步事件总线）
   - Object Storage（证据附件、审计归档）

## 2.2 核心链路（报平安）
客户端点击“我还活着” → API 网关鉴权 → Check-in 服务写库 → 发布 `alive_checkin_success` 事件 → Reminder 服务重置倒计时 → Feed 服务写入好友动态 → Notification 服务可选发送“已恢复”通知。

## 2.3 核心链路（失联提醒）
定时任务扫描/延迟队列触发 → Reminder 服务判断阈值与免打扰 → 先发本人提醒 → 达到升级条件后调用 Notification 服务通知紧急联系人 → 写入审计日志与告警事件。

## 3. 领域模型与数据表设计

## 3.1 核心实体
- User（用户）
- AliveRecord（报平安记录）
- ReminderRule（提醒规则）
- ReminderIncident（提醒事件）
- EmergencyContact（紧急联系人）
- Friendship（好友关系）
- FeedPost / FeedInteraction（动态与互动）
- AbuseReport（举报）

## 3.2 表结构（示例）

### `users`
- `id` BIGINT PK
- `phone_hash` VARCHAR(128)
- `nickname` VARCHAR(32)
- `status` TINYINT（active/disabled/deleted）
- `created_at` DATETIME
- `updated_at` DATETIME

### `alive_records`
- `id` BIGINT PK
- `user_id` BIGINT INDEX
- `template_code` VARCHAR(32)
- `content` VARCHAR(200)
- `visibility` TINYINT（self/friends/group）
- `checkin_at` DATETIME INDEX
- `source` VARCHAR(16)（manual/widget/api）

### `reminder_rules`
- `id` BIGINT PK
- `user_id` BIGINT UNIQUE
- `threshold_hours` INT
- `grace_minutes` INT
- `dnd_start` TIME
- `dnd_end` TIME
- `escalation_enabled` BOOLEAN
- `updated_at` DATETIME

### `reminder_incidents`
- `id` BIGINT PK
- `user_id` BIGINT INDEX
- `status` TINYINT（pending/notified_self/notified_emergency/resolved/false_alarm）
- `trigger_at` DATETIME
- `resolved_at` DATETIME NULL
- `notify_count` INT
- `idempotency_key` VARCHAR(64) UNIQUE

### `emergency_contacts`
- `id` BIGINT PK
- `user_id` BIGINT INDEX
- `contact_name` VARCHAR(32)
- `contact_phone_enc` VARBINARY(512)
- `authorized_at` DATETIME
- `is_verified` BOOLEAN

## 4. API 设计（MVP）

## 4.1 鉴权
- JWT + Refresh Token。
- 所有写接口要求 `X-Device-Id` 与签名头，防刷与审计。

## 4.2 关键接口
1. `POST /v1/checkins`
   - 入参：`templateCode`, `content`, `visibility`, `silent`
   - 出参：`checkinId`, `nextDeadlineAt`
2. `GET /v1/checkins/latest`
   - 出参：最近报平安记录 + 剩余倒计时
3. `PUT /v1/reminder-rule`
   - 入参：`thresholdHours`, `graceMinutes`, `dndWindow`, `escalationEnabled`
4. `POST /v1/emergency-contacts`
   - 入参：`name`, `phone`, `verificationCode`
5. `GET /v1/feed`
   - 出参：好友动态列表（分页游标）
6. `POST /v1/reports`
   - 入参：`targetType`, `targetId`, `reason`, `evidenceUrls`

## 4.3 错误码策略
- `4001` 参数非法
- `4003` 无权限（被拉黑/不可见）
- `4091` 幂等冲突
- `4290` 触发频率限制
- `500x` 内部错误（带 traceId）

## 5. 提醒调度与状态机设计

## 5.1 状态机
`IDLE` → `NEAR_DEADLINE` → `OVERDUE_SELF_NOTIFIED` → `OVERDUE_ESCALATED` → `RESOLVED` / `FALSE_ALARM`

## 5.2 调度策略
- 采用“延迟队列 + 定时补偿扫描”双机制。
- 关键字段：`next_trigger_at`、`version`（乐观锁）。
- 任何通知发送前校验 incident 状态，确保幂等。

## 5.3 频控策略
- 本人提醒：每 2 小时最多 1 次。
- 紧急联系人提醒：每事件最多 1 次，每 24h 最多 1 次。
- 催更行为：单好友每天最多 3 次。

---

## 6. 隐私、安全与风控设计

## 6.1 数据安全
- 手机号、联系人电话使用 KMS 管理密钥加密。
- 传输层全站 TLS1.2+，服务间 mTLS（内网）。
- 审计日志脱敏存储，默认保留 180 天。

## 6.2 访问控制
- RBAC：普通用户、审核员、运营管理员。
- 审核后台所有敏感操作必须二次确认与操作留痕。

## 6.3 风控策略
- 注册/登录：设备指纹 + 行为评分拦截批量注册。
- 互动内容：敏感词 + 模型分级审核。
- 通知滥发：规则变更冷却期（如 30 分钟）防止恶意触发。

---

## 7. 可观测性与运维

## 7.1 指标（Metrics）
- `checkin_qps`, `checkin_success_rate`
- `reminder_trigger_delay_p95`
- `notification_delivery_rate`
- `false_alarm_rate`
- `report_sla_hours`

## 7.2 日志与追踪
- 全链路 traceId（网关注入）。
- 关键行为日志结构化（JSON）。
- 告警规则：提醒延迟、通知失败率、数据库慢查询。

## 7.3 灰度与回滚
- 按用户分桶灰度（5%/20%/50%/100%）。
- 关键配置中心化管理，可 1 分钟内回滚。

---

## 8. 客户端设计要点

## 8.1 首页交互
- 顶部显示“距下次提醒还剩 xx:xx”。
- 中央主按钮固定位置，支持单手操作。
- 报平安成功后即时反馈（动效 + 轻振动）。

## 8.2 离线与弱网
- 本地持久化最近一次点击事件。
- 网络恢复后按时间顺序补发，避免重复（幂等 token）。

## 8.3 推送策略
- 常规互动：可合并摘要推送。
- 超时提醒：高优先级独立推送，不与营销消息混发。

---

## 9. 测试与验收方案

## 9.1 测试类型
- 单元测试：规则计算、状态机流转、幂等校验。
- 集成测试：Check-in → Reminder → Notification 全链路。
- 压测：高峰 5k QPS 报平安写入，提醒延迟与丢失率评估。
- 安全测试：越权访问、注入、短信轰炸、重放攻击。

## 9.2 关键验收用例
1. 用户在阈值前 5 分钟报平安，不触发超时提醒。
2. 用户超时后本人提醒发送成功，恢复报平安后 incident 置 `RESOLVED`。
3. 免打扰时段内不升级通知，时段结束后补发策略符合配置。
4. 同一 incident 多次消费不重复通知（幂等验证）。

---

## 10. 迭代路线建议
- **MVP**：单人提醒 + 好友流 + 基础风控。
- **V1.1**：群组、徽章、桌面小组件。
- **V1.2**：AI 文案建议、异常模式识别（仅提醒，不诊断）。
- **V2.0**：开放 API（第三方设备/日历集成）与国际化。
